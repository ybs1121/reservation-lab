# Reservation Hold 확정-해제 동시성 재현 계획

## 목표

같은 hold에 확정과 해제가 동시에 요청될 때 현재 구현에서 두 작업이 모두 성공할 수 있는 경쟁 조건을 테스트로 재현한다.

## 정책

- 확정과 해제 중 락을 먼저 획득한 작업 하나만 성공해야 한다.
- 나중 작업은 이미 삭제된 hold를 확인하고 `RESERVATION_HOLD_NOT_FOUND`로 실패해야 한다.

## 테스트 재현 순서

1. Redis에 유효한 hold를 저장한다.
2. 확정 요청이 락 안에서 hold를 재조회한 뒤 예약 생성 직전에 대기하게 한다.
3. 그 사이 현재 락이 없는 해제 요청을 실행해 hold를 삭제한다.
4. 확정 요청을 재개한다.
5. 두 요청 중 성공이 정확히 하나인지 검증한다.

## 이번 범위

- 1차로 실패 재현 테스트를 추가한다.
- 재현 확인 후 확정과 해제에 동일한 사용자·슬롯 락을 적용한다.
- 해제는 락 획득 후 hold를 다시 조회해 선행 확정 여부를 확인한다.

## 수용 기준

- 현재 코드에서 테스트가 `expected: 1, actual: 2`로 실패해 경쟁 조건을 증명한다.

## 재현 결과

- Redis Testcontainers 환경에서 테스트를 실행했다.
- 사용자 hold 최대 3개 다중 락 테스트는 통과했다.
- 확정-해제 동시성 테스트는 `expected: 1, actual: 2`로 의도대로 실패했다.
- 확정이 hold를 재검증한 뒤에도 락 없는 해제가 같은 hold를 삭제할 수 있어 두 요청이 모두 성공함을 확인했다.

## 수정 방향

- 확정과 해제 모두 사용자 락과 슬롯 락을 획득한다.
- 해제 로직은 별도 `ReservationHoldReleaseService`로 분리해 AOP 프록시가 적용되게 한다.
- 최초 hold 조회는 락 키 확보에만 사용하고, 해제 서비스가 락 안에서 hold를 다시 조회한다.

## 수정 결과

- 확정과 해제가 동일한 사용자 락과 슬롯 락을 획득하도록 변경했다.
- 확정이 먼저 락을 획득한 재현 시나리오에서 확정만 성공하고, 뒤의 해제는 `RESERVATION_HOLD_NOT_FOUND`로 실패한다.
- 수정 전 `expected: 1, actual: 2`로 실패했던 Redis 동시성 테스트가 통과했다.
- 사용자별 동시 hold 최대 3개 테스트도 함께 통과했다.
- 해제 서비스의 hold 재조회와 소유자 검증 단위 테스트가 통과했다.
