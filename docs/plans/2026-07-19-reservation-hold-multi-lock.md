# Reservation Hold 다중 분산락 적용 계획

## 목표

같은 사용자가 서로 다른 슬롯에 동시에 hold를 생성할 때 사용자별 active hold 최대 3개 정책이 깨지는 경쟁 조건을 재현하고 해결한다.

## 현재 문제

- hold 생성은 슬롯 단위 분산락만 사용한다.
- 같은 사용자가 서로 다른 슬롯에 요청하면 서로 다른 슬롯 락을 획득하므로 동시에 처리된다.
- 각 요청이 사용자 active hold 개수를 같은 시점에 조회하면 최대 3개를 초과해 저장할 수 있다.

## 적용 방향

- 기존 `@DistributedLock`이 N개의 SpEL 키를 받을 수 있도록 확장한다.
- 단일 키를 사용하는 기존 예약 생성 및 hold 확정 코드는 현재 사용법을 유지한다.
- 해석된 락 키는 중복을 제거하고 정렬해 모든 호출 경로에서 획득 순서를 동일하게 만든다.
- Redisson `MultiLock`으로 여러 락의 획득과 해제를 한 묶음으로 관리한다.
- hold 생성은 아래 순서로 락을 획득한다.
  1. `lock:reservation-hold:user:{userId}`
  2. `lock:reservation:slot:{slotId}`
- 모든 락을 획득한 뒤 기존 hold 조회, 사용자 제한 검증, 슬롯 capacity 검증, Redis 저장을 실행한다.
- 락 획득에 실패하면 이미 정의된 `LOCK_ACQUIRE_FAILED` 정책을 유지한다.

## 범위

- `DistributedLock` N개 키 지원
- `DistributedLockAspect` 다중 락 획득 및 해제
- `ReservationHoldService#createHold`에 사용자 락과 슬롯 락 적용
- 같은 사용자의 서로 다른 슬롯 동시 hold 생성 테스트
- 기존 단일 락 동작 회귀 테스트

## 제외 범위

- Redis Lua 기반 원자 저장
- RabbitMQ 중복 소비 처리
- hold 확정과 해제의 락 범위 변경
- 락 대기 시간 및 lease time 정책 변경

## 수용 기준

- 동일 사용자가 서로 다른 슬롯에 동시에 hold를 요청해도 active hold는 최대 3개다.
- 동일 슬롯의 hold 합계는 capacity를 초과하지 않는다.
- 기존 예약 생성의 단일 슬롯 락이 정상 동작한다.
- 관련 테스트가 통과한다.

## 구현 후 확인할 학습 포인트

- 락 키가 하나일 때와 여러 개일 때의 획득 과정 차이
- 락 획득 순서를 고정하지 않을 때 발생할 수 있는 교착 상태
- 사용자 단위 직렬화가 서로 다른 슬롯 요청의 처리량에 주는 영향
- 다중 락과 Redis Lua 원자 연산의 장단점

## 결정 기록

### 2026-07-19

- 두 개 전용 어노테이션을 추가하지 않고 기존 `DistributedLock`을 N개 키 배열 방식으로 확장한다.
- 호출자가 배열 순서를 다르게 작성해도 교착 가능성을 낮출 수 있도록 실제 락 키를 정렬한 뒤 획득한다.

## 결과

- `DistributedLock.keys`가 단일 키와 N개 키를 모두 받을 수 있도록 변경했다.
- AOP는 실제 락 키의 중복을 제거하고 정렬한 뒤, 여러 키일 때 Redisson `MultiLock`을 사용한다.
- hold 생성은 사용자 락과 슬롯 락을 함께 획득하므로 서로 다른 슬롯 요청도 사용자 단위로 직렬화된다.
- 다중 키 정렬 및 단일 키 회귀 단위 테스트가 통과했다.
- 동일 사용자의 서로 다른 슬롯 4개 동시 hold 테스트를 추가했다. 현재 로컬 Docker 데몬이 실행 중이 아니어서 이 통합 테스트는 건너뛰어졌으며, Docker 실행 환경에서 최종 확인이 필요하다.
