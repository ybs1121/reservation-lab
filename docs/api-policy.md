# api-policy.md

이 프로젝트의 API는 HTTP Method, 공통 응답 형식, 상태코드, 예외 처리 방식을 단순하고 일관되게 사용한다.

## HTTP Method 원칙
- GET: 조회
- POST: 생성
- PUT 또는 PATCH: 수정
- DELETE: 삭제

## 응답 구조
성공 응답은 아래 공통 형식을 사용한다.

### 성공
```json
{
  "success": true,
  "data": {},
  "message": null,
  "code": null
}
```

실패 응답은 아래 에러 형식을 사용한다.

### 실패
```json
{
  "success": false,
  "message": "에러 메시지",
  "code": "USR00001"
}
```
## 상태코드 원칙

기본적으로 아래 상태코드를 사용한다.

- 200: 조회/생성/수정/삭제 성공
- 400: 잘못된 요청 또는 예상 가능한 비즈니스 규칙 위반
- 500: 예상하지 못한 서버 오류

특별한 경우가 아니면 상태코드를 과하게 세분화하지 않는다.

## 예외 처리 원칙
- 예상 가능한 비즈니스 예외는 BizException으로 처리한다
- BizException은 공통 예외 처리 구조를 사용한다
- BizException 발생 시 상태코드는 400을 사용하고 실패 응답 body를 반환한다
- 예상하지 못한 예외는 500으로 처리한다

## 메시지 원칙
- 사용자에게 노출할 에러 메시지는 messages.properties에 정의한다
- 에러 메시지는 한글로 관리한다
- 코드에서 메시지 문자열을 직접 하드코딩하지 않는다
- 에러 코드 원칙
- BizException의 에러 코드는 8자리로 제한한다. (영어 대무나 3글자 + 숫자 5글자)
- 에러 코드는 도메인의 약어를 만들어서 넣는다. (User일 경우 USR)
- 에러 코드는 프로젝트 내에서 중복되지 않도록 관리한다
- 에러 코드는 messages.properties의 키와 1:1로 매핑되도록 관리한다
- 에러 코드의 접두어 3자리는 도메인 별로 고유해야 한다. 다만 공통과 관련 없이 경우에는 COM 으로 통일한다.

## 추가 원칙
- 성공 응답의 data는 DTO를 사용한다
- Entity를 API 응답으로 직접 반환하지 않는다
- 문서에 없는 응답 구조는 임의로 추가하지 않는다
