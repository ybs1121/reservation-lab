# project-structure.md

이 프로젝트는 도메인 기준으로 패키지를 나눈다.

## 기본 구조

각 도메인은 아래와 같은 하위 패키지 구조를 따른다.

- controller
- service
- repository
- entity
- dto
- component

예시:

<domain>
├── controller
├── service
├── repository
├── entity
├── dto
└── component

각 공통은 아래와 같은 하위 패키지 구조를 따른다.

<common>
├── config
├── component

추가적으로 필요한 경우가 있으면 사용자에게 물어 본 후 구조를 추가한다.

## 역할

### controller
- HTTP 요청/응답 처리
- 요청 파라미터와 응답 DTO 처리
- 비즈니스 로직은 직접 수행하지 않는다

### service
- 유스케이스와 비즈니스 로직 처리
- 여러 객체를 조합하는 흐름 담당

### repository
- 데이터 저장/조회 담당

### entity
- 도메인 상태와 영속성 매핑 객체 위치

### dto
- 요청/응답 데이터 객체 위치
- Entity를 API 응답으로 직접 노출하지 않는다

### component
- 도메인 내부 보조 역할
- 변환, 조회 보조, 검증 보조 등
- 핵심 비즈니스 로직은 service에 둔다

## 원칙
- 새로운 도메인이 추가되어도 동일한 구조를 우선 따른다
- 구조가 필요 이상으로 복잡해지지 않도록 한다
- 문서에 없는 새로운 레이어나 패키지는 임의로 추가하지 않는다
- 구조 변경이 필요하면 먼저 변경 이유를 드러낸다
- 요청된 기능 중 정책 미정으로 보류되는 부분은 흔적 없이 지우지 않고 TODO로 남겨 추적 가능하게 한다
- getter, setter, 생성자 등 반복 보일러플레이트는 Lombok을 우선 활용한다
- enum이나 핵심 비즈니스 로직에는 의도를 빠르게 이해할 수 있는 짧은 주석을 남긴다
- 단순 getter/setter 수준이 아닌 메서드는 역할이나 의도를 바로 알 수 있도록 필요한 경우 짧은 주석을 남긴다
- 주석은 코드가 무엇을 하는지 반복하지 않고, 왜 필요한지 또는 어떤 유스케이스 흐름을 담당하는지 설명한다
- Entity를 응답 DTO로 치환하는 책임은 service에 두지 않고, 응답 DTO의 정적 메서드로 둔다

## 설계 방향
이 프로젝트는 Lite DDD를 지향한다.

- 도메인 기준으로 패키지를 나눈다
- 각 도메인의 상태와 규칙이 코드에 드러나도록 한다
- 작은 프로젝트에 맞게 단순하고 읽기 쉬운 구조를 우선한다
- 복잡한 계층 분리, 과도한 추상화, 불필요한 DDD 패턴은 도입하지 않는다

## 도메인 공통정책
- 모든 도메인은 작성일자, 작성자, 수정일자, 수정자를 가진다.
- 작성자, 수정자는 user_id를 기준으로 하며, 만약 user_id가 나올 수 없는 경우 예를 들어서 org 등록과 같은 경우에는 SYSTEM 이라고 작성한다
