## 문서 참조 기준

- 도메인 정책, 상태값, 삭제 정책이 필요하면 docs/domain-rules.md를 본다
- 패키지 구조, 계층 역할이 필요하면 docs/project-structure.md를 본다
- 이전에 반복된 실수나 주의사항은 docs/known-pitfalls.md를 본다
- 작업 요청 형식, 수용 기준, 테스트 기준은 docs/task-template.md를 본다
- HTTP Method, 공통 응답 형식, 상태코드, BizException 정책은 docs/api-policy.md를 참고한다

## 기본 원칙
- 현재 브랜치가 master면 코드를 수정하기 전에 사용자에게 한 번 확인한다
- Controller는 요청/응답 처리만 담당한다
- Service는 유스케이스와 비즈니스 규칙을 담당한다
- Repository는 DB 접근만 담당한다
- Entity를 API 응답으로 직접 반환하지 않는다
- Entity → DTO 변환 책임은 service에 두지 않고, 응답 객체의 정적 메서드로 둔다
- 이 프로젝트는 Lite DDD 관점으로 구현한다
- 도메인 기준으로 구조를 나누고, 상태와 규칙이 코드에 드러나도록 한다
- 단순 CRUD 수준의 단계에서는 과한 계층 분리나 복잡한 DDD 패턴은 도입하지 않는다
- 문서에 정의되지 않은 정책이나 구조는 임의로 추가하지 않는다
- 정책이 모호하거나 비어 있으면 먼저 사용자에게 확인한다
- 사용자가 정책을 미정으로 두면 TODO 또는 결정 필요 항목으로 남긴다
- 사용자가 요청한 기능을 정책 미정으로 되돌리더라도, 로직을 완전히 제거하기보다 가능한 범위는 유지하고 제외된 부분은 TODO로 남긴다
- 기능 변경 시 관련 테스트를 함께 수정하거나 추가한다
- getter, setter, 생성자 등 반복 보일러플레이트는 Lombok을 우선 활용한다
- enum이나 핵심 비즈니스 로직처럼 의미 해석이 필요한 지점에는 짧은 주석을 남긴다
- 단순 getter/setter 수준이 아닌 메서드는 역할이나 의도를 바로 알 수 있도록 필요한 경우 짧은 주석을 남긴다
- 주석은 코드가 무엇을 하는지 반복하지 않고, 왜 필요한지 또는 어떤 유스케이스 흐름을 담당하는지 설명한다

## 작업 계획 원칙

- 작업 범위가 크거나 정책 판단이 필요한 경우 `docs/plans/` 아래에 작업 계획과 결과를 기록한다
- 단순 CRUD, 단순 문구 수정, 사소한 리팩터링이 아니면 먼저 계획을 작성하고 사용자 확인 후 실행한다
- 사용자가 진행 과정이나 판단 근거를 확인할 수 있도록, 필요한 경우 계획 문서를 먼저 작성하거나 갱신한다
- 단순 CRUD, 단순 문구 수정, 사소한 리팩터링은 계획 문서를 생략할 수 있다

## 실수 기록 원칙

- 재발 가능성이 있는 실수나 주의사항은 docs/known-pitfalls.md에 기록한다
- 이미 기록된 실수/주의사항이 있으면 작업 전에 참고한다
- 일회성 오타나 사소한 수정 내역은 기록하지 않는다

# Working Principles

AI와 함께 작업할 때 따르는 기본 행동 지침. 속도보다 신중함에 가중치를 둔다. 사소한 작업에는 판단해서 적용하지 않아도 된다.

## 1. Think Before Coding

**추측하지 말고, 헷갈리면 멈춰서 묻는다.**

구현 전에:

- 가정은 명시한다. 불확실하면 질문한다.
- 해석이 여러 가지 가능하면 조용히 하나를 고르지 말고 제시한다.
- 더 단순한 접근이 있으면 말한다. 필요하면 반박한다.
- 명확하지 않으면 멈춘다. 무엇이 헷갈리는지 말하고 묻는다.

## 2. Simplicity First

**요청한 것을 푸는 최소 코드. 추측성은 없다.**

- 요청받지 않은 기능은 넣지 않는다.
- 한 번 쓰는 코드에 추상화를 만들지 않는다.
- 요청하지 않은 "유연성", "설정 가능성"을 넣지 않는다.
- 일어나지 않을 시나리오에 대한 예외 처리를 넣지 않는다.
- 200줄로 짠 게 50줄로 가능하다면 다시 쓴다.

스스로 묻기 — "시니어가 보면 과설계라고 할까?" 그렇다면 단순화한다.

## 3. Surgical Changes

**꼭 필요한 것만 건드린다. 자기가 만든 것만 정리한다.**

기존 코드를 수정할 때:

- 인접한 코드·주석·포맷을 "개선"하지 않는다.
- 망가지지 않은 것을 리팩토링하지 않는다.
- 다르게 쓰고 싶더라도 기존 스타일을 따른다.
- 관련 없는 죽은 코드는 언급만 한다. 삭제하지 않는다.

내 변경이 만들어낸 잔여물:

- 내 변경 때문에 안 쓰이게 된 import·변수·함수만 제거한다.
- 원래 있던 죽은 코드는 요청이 없으면 건드리지 않는다.

기준 — 변경된 모든 줄은 사용자의 요청과 직접 연결되어야 한다.

## 4. Goal-Driven Execution

**성공 기준을 정의하고, 검증될 때까지 반복한다.**

작업을 검증 가능한 목표로 변환한다:

- "검증 추가" → "잘못된 입력에 대한 테스트 작성 후 통과시키기"
- "버그 수정" → "버그를 재현하는 테스트 작성 후 통과시키기"
- "X 리팩토링" → "변경 전후 모두 테스트 통과 보장"

여러 단계의 작업이면 짧은 계획을 먼저 제시한다:

```
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
3. [단계] → 검증: [확인 방법]
```

명확한 성공 기준이 있으면 독립적으로 반복할 수 있다. 약한 기준("동작하게")은 계속 추가 질문을 만든다.
